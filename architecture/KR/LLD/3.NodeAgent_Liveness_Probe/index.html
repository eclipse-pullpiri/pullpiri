    NodeAgent Self-Healing - Liveness Probe¶

    0. 문서의 목적¶

    • 목표: Liveness probe로 running 컨테이너의 건강 상태를 확인하고 3회 연속 실패 시 컨테이너를 stop하는 기능
    • 왜 필요한가: 컨테이너가 running 상태이지만 응답이 없는 경우(dead lock, hang 등)를 감지하여 재시작이 필요함
    • 이 문서는 NodeAgent의 manager.rs에 probe_loop 함수와 probe/ 디렉터리의 liveness probe 기능을 추가하기 위해 작성되었습니다.
    • 이 기능은 이 문서에 포함된 조건 및 규칙들을 따라야 합니다.

    1. Probe의 기능¶

    • Liveness Probe: 컨테이너가 살아있는지 확인 (HTTP/TCP/Exec)
    • 연속 실패 카운트: 3회 연속 실패 시 컨테이너 stop

    2. Probe의 구현 구조¶

    • manager.rs: probe_loop() 실행
    • probe/liveness.rs: LivenessProbe 구조체, check_liveness_probe() 함수
    • probe/checker.rs: HTTP/TCP/Exec 프로브 실행 로직
    • runtime/podman/container.rs: stop() 함수 (기존)

    2.1. Pod YAML 파싱 요구사항 (⚠️ CRITICAL)¶

    Liveness Probe가 동작하려면 반드시 Pod YAML의 probeConfig를 파싱하여 DesiredState에 저장해야 합니다.

    필수 구현 파일¶

    src/common/src/spec/k8s/pod.rs¶

    PodSpec에 probeConfig 필드 추가:

    pub struct PodSpec {
    // ... 기존 필드들 ...
    pub probeConfig: Option<ProbeConfig>,
}

pub struct ProbeConfig {
    pub liveness: Option<LivenessProbe>,
}

pub struct LivenessProbe {
    pub http: Option<HttpProbe>,
    pub tcp: Option<TcpProbe>,
    pub exec: Option<ExecProbe>,
    pub initialDelaySeconds: u32,
    pub periodSeconds: u32,
    pub timeoutSeconds: u32,
    pub failureThreshold: u8,
}

pub struct HttpProbe { pub path: String, pub port: u16 }
pub struct TcpProbe { pub port: u16 }
pub struct ExecProbe { pub command: Vec<String> }

    구현하지 않으면 발생하는 문제¶

    • YAML에 probeConfig 명시해도 무시됨
    • probe_config가 항상 None → Probe 실행 안 됨
    • journalctl에서 Probe 로그 보이지 않음

    3. Liveness Probe를 위해 Probe 모듈에 구현되어야 하는 것¶

    +-------------------+         +---------------------+         +-------------------+
|   Podman          | inspect |   NodeAgent         | HTTP/   |   Container       |
| (running 목록)    | ------> | (probe_loop)        | TCP/    | (health check)    |
|                   |         |                     | Exec    |                   |
+-------------------+         +---------------------+ ------> +-------------------+
                                        |
                                        | 3회 연속 실패 시
                                        v
                              podman::stop(container_id)
    • 인터페이스: Podman API 조회 및 호출, HTTP/TCP/Exec 엔드포인트
    • 입력:
      1. Podman에서 running 상태 컨테이너 목록 조회
      2. 메모리 캐시(desired_states_cache)에서 각 컨테이너의 desired_state 조회
      3. ⚠️ 중요: desired_state의 probe_config는 actioncontroller.rs에서 Pod YAML 파싱 시 채워져야 함
      4. probe_config가 None이면 해당 컨테이너는 Probe 실행 대상이 아님
    • 처리:
      1. 매 1초마다 실행 (메인 루프)
      2. Running 컨테이너마다 조건 확인:
      3. initial_delay_seconds 경과 확인 (컨테이너 시작 시간 기준)
      4. period_seconds 간격 확인 (마지막 probe 시간 기준)
      5. Probe 조건 충족 시 liveness probe 실행
      6. 실패 시 failure_count 증가 (메모리에 HashMap으로 관리)
      7. 성공 시 failure_count 초기화
      8. failure_count >= failure_threshold이면 podman stop 호출
    • 출력: failure_threshold 이상 연속 실패 시 컨테이너 stop

    4. 지켜야 할 규칙¶

    • 실행 주기: 1초마다 메인 루프 실행 (각 probe는 period_seconds 간격으로 실행) 
      pub async fn probe_loop(desired_states_cache: Arc<Mutex<HashMap<String, DesiredState>>>) {
    let mut probe_states: HashMap<String, ProbeState> = HashMap::new();

    loop {
        // 1. Running 컨테이너 조회
        // 2. 각 컨테이너의 probe 설정 확인
        // 3. initial_delay_seconds 및 period_seconds 확인 후 probe 실행
        tokio::time::sleep(Duration::from_secs(1)).await;
    }
}

struct ProbeState {
    failure_count: u8,
    container_started_at: SystemTime,  // 컨테이너 시작 시간
    last_probe_at: Option<SystemTime>, // 마지막 probe 실행 시간
}
    • Probe 타입: HTTP, TCP, Exec 세 가지 지원
    • ProbeType::Http { path, port }: HTTP GET 요청
    • ProbeType::Tcp { port }: TCP 연결 시도
    • ProbeType::Exec { command }: 컨테이너 내부 명령어 실행
    • 실패 판단:
    • HTTP: 응답 코드 200-399가 아니면 실패
    • TCP: 연결 실패 시 실패
    • Exec: exit code 0이 아니면 실패
    • 상태 관리:
    • failure_counts: 메모리에 HashMap으로 관리 (HashMap<String, u8>)
    • probe_states: 메모리에 HashMap으로 관리 (타이밍 체크용)
    • 타이밍 체크:
    • initial_delay_seconds: 컨테이너 시작 후 첫 probe까지 대기
    • period_seconds: probe 실행 간격
    • timeout_seconds: 각 probe 실행의 타임아웃
    • 에러 처리: probe 실행 실패 시 failure_count 증가
    • Failure Threshold: desired_state의 probe_config에서 지정된 값 사용 (기본값 3)
    • 메모리 캐시 접근: desired_states_cache에서 DesiredState 조회

    5. 구현 로직 (의사코드)¶

    probe_loop 흐름¶

    초기화:
  failure_counts = HashMap::new()  // container_id → failure_count
  probe_states = HashMap::new()    // container_id → ProbeState

매 1초마다 반복:
  FOR EACH running_container:
    1. 메모리 캐시(desired_states_cache)에서 desired_state 조회
       - probe_config 없으면 SKIP
       - liveness probe 설정 없으면 SKIP

    2. 타이밍 체크:
       - Podman inspect로 컨테이너 시작 시각 조회
       - IF (현재시각 - 컨테이너시작시각) < initial_delay_seconds:
           SKIP (아직 초기 대기 중)
       - IF (현재시각 - 마지막probe시각) < period_seconds:
           SKIP (다음 probe 주기 아님)

    3. Probe 실행 (timeout_seconds 내에):
       - HTTP: GET http://localhost:PORT/PATH
           → 200-399 응답이면 성공
       - TCP: connect 127.0.0.1:PORT
           → 연결 성공하면 성공
       - Exec: podman exec CONTAINER_ID COMMAND
           → exit_code == 0이면 성공

    4. 결과 처리:
       IF 성공:
         failure_counts[container_id] = 0 (초기화)
       ELSE:
         failure_counts[container_id]++

         IF failure_counts[container_id] >= failure_threshold:
           podman stop container_id
           failure_counts.remove(container_id)
           로그: "Liveness probe failed N times, stopping container"

    5. probe_states 업데이트:
       last_probe_at = 현재시각

    핵심 데이터 구조¶

    // ProbeState: 타이밍 추적용
struct ProbeState {
    container_started_at: SystemTime,  // Podman inspect에서 조회
    last_probe_at: Option<SystemTime>,
}

// 메모리 관리
let failure_counts: HashMap<String, u8> = HashMap::new();
let probe_states: HashMap<String, ProbeState> = HashMap::new();

    컨테이너가 제공해야 하는 것 (별도 에이전트 불필요)¶

    • HTTP Probe: 컨테이너 내부에서 HTTP 서버 실행 (예: nginx /health, web app /ready)

    • 기존 서버의 health endpoint 활용 - 추가 설치 불필요

    • TCP Probe: 컨테이너 내부에서 TCP 포트 리슨 (예: MySQL 3306, Redis 6379)

    • 서비스가 정상 실행 중이면 자동으로 포트 오픈 - 추가 작업 불필요

    • Exec Probe: 컨테이너 내부에서 실행 가능한 명령어 제공

    • 예: cat /tmp/healthy, pgrep myapp, curl localhost:8080/health
    • 컨테이너 내부 명령어만 있으면 됨 - 별도 에이전트 불필요

    구현 시 주의사항¶

    • Podman API로 컨테이너 상태 조회: GET /v4.0.0/libpod/containers/json?filters={"status":["running"]}
    • Podman API로 컨테이너 중지: POST /v4.0.0/libpod/containers/{id}/stop
    • Podman exec 실행: POST /v4.0.0/libpod/containers/{id}/exec (Exec Probe용)
    • HTTP 요청 시 timeout 적용 필수 (무한 대기 방지)
    • probe_states는 컨테이너 삭제 시 정리 필요 (메모리 누수 방지)

    5.1. 구현 검증 방법¶

    구현 완료 후 반드시 다음 테스트를 통과해야 합니다:

    # 1. nginx 배포 (probeConfig 포함된 YAML 사용)
./examples/nginx.sh

# 2. Probe 로그 확인
journalctl -u nodeagent -f | grep -i probe

# 예상 출력 (periodSeconds=10초 간격):
# [Probe] Checking liveness probe for container nginx
# [Probe] Liveness probe for container nginx: HTTP GET / on port 80 - Success

    동작 확인: - ✅ probeConfig가 YAML에 있으면: Probe 실행, 로그 출력 - ✅ probeConfig가 YAML에 없으면: Probe 실행 안 함 (정상) - ✅ 3회 연속 실패 시: 컨테이너 stop 로그 출력

    실패 테스트: 

    # nginx 프로세스 강제 종료 (컨테이너는 running 상태 유지)
podman exec nginx pkill nginx

# 예상: 3회 probe 실패 후 컨테이너 stop
# [Probe] Liveness probe failed (1/3)
# [Probe] Liveness probe failed (2/3)
# [Probe] Liveness probe failed (3/3)
# [NodeAgent] Stopping container nginx due to liveness probe failure
